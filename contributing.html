Contributing

How to plan, design, develop, test, and deploy features in ClaudeCockpit.
See also: Roadmap · Changelog · Testing Pipeline

How We Work

ClaudeCockpit is a personal tool built by one developer (Bryan) + Claude Code. There's no team, no sprint board, no PR review queue. The contributor model is conversational: you describe what you want, Claude Code implements it, you verify and ship.

1
Design for Claude Code, not OpenClaw. Borrow infrastructure patterns (Lit, WS protocol, CSS tokens) but build the information architecture around Claude Code's concepts: projects, subagents, plan mode, tasks, permissions, memory.
2
Personal tool, not a product. No themes, i18n, onboarding, multi-user, or feature flags. Lean and opinionated for a single power user on macOS.
3
Read real data first, control later. Understand the data shape before building controls on top of it. Don't over-engineer before knowing what you're working with.

1. Plan

Before writing code, know what you're building and why.

Where to look

  • roadmap.html — open the Phases section. Items with check--todo are uncompleted work. The Feature Backlog has lower-priority ideas.
  • Design Decisions Log (bottom of roadmap) — past architectural decisions with rationale. Check here before making a decision that contradicts an existing one.
  • git log --oneline -20 — recent work gives context on what's in-flight.

Scoping a feature

Ask yourself: Does this touch the gateway, the UI, or both? If both, which layer owns the complexity? A feature that's "mostly a new service with a thin UI" is different from "mostly a new UI view that calls an existing endpoint."

The type boundary

Types are intentionally duplicated across two files. This is a deliberate architectural boundary.

gateway/types.ts
Models Claude Code's on-disk format and API response shapes.
Add here: new JSONL fields, raw content block types, API response types.
Examples: RawSessionLine, RawContentBlock, RawUsage, SessionSummary
ui/types.ts
Simplified shapes for rendering, plus UI-only state.
Add here: component prop types, rendering state, client-side event types.
Examples: ChatMessage (adds streaming), ToolApprovalEvent

Data flow: end-to-end

Gateway Service Server (WS) UI Client UI Component ────────────── ─────────── ───────── ──────────── Pure function → wsHandler calls it → gateway.request() → @property data in services/ sends okResponse() returns Promise triggers render() or eventFrame() or on() callback
HTTP vs WebSocket. Use HTTP (/api/*) for one-shot reads the UI needs at load time (overview stats, session list). Use WebSocket for anything that streams, pushes events, or needs low-latency request-response (chat, tool approval, live updates).

if (path === "/api/my-feature") {

WebSocket Method

gateway/server.ts Write a handler function, add it to the wsHandlers dispatch table.

// 1. Handler function
async function handleMyFeature({ ws, id, p }: WsContext) {
  const name = requireString(p, "name");
  if (!name) {
    send(ws, errResponse(id, "INVALID_PARAMS", "name is required"));
    return;
  }
  const result = await myServiceFunction(name);
  send(ws, okResponse(id, result));
}

// 2. Register in the dispatch table
const wsHandlers = {
  // ... existing handlers ...
  "myFeature.do": handleMyFeature,
};

Pushing Events (server → client)

import { eventFrame } from "../protocol/frames.ts";

// Push to one client
send(ws, eventFrame("myFeature.update", { progress: 42 }));

// Push to all connected clients
for (const client of wss.clients) {
  if (client.readyState === WebSocket.OPEN) {
    send(client, eventFrame("myFeature.update", data));
  }
}

UI Component

ui/views/ Lit web components in light DOM. All styling via global CSS.

// ui/views/my-feature.ts
import { LitElement, html, nothing } from "lit";
import { customElement, property, state } from "lit/decorators.js";

@customElement("cockpit-my-feature")
export class CockpitMyFeature extends LitElement {
  protected override createRenderRoot() { return this; }

  @property({ type: Object }) data: MyData | null = null;
  @state() private _loading = false;

  override connectedCallback() {
    super.connectedCallback();
    // Subscribe to events, start timers
  }
  override disconnectedCallback() {
    super.disconnectedCallback();
    // Clean up subscriptions, timers
  }

  override render() {
    if (!this.data) return html`<div class="skeleton"></div>`;
    return html`
      <section class="my-feature">
        <h2>${this.data.title}</h2>
        ${this._loading ? html`<span>Loading...</span>` : nothing}
      </section>
    `;
  }
}
1
Light DOM always. createRenderRoot() { return this; } — global CSS applies. No Shadow DOM.
2
@property for inputs, @state for internal. Both trigger re-renders. Properties are set by parent; state is private.
3
Use nothing not empty strings. ${condition ? html`...` : nothing} removes content entirely.
4
Clean up in disconnectedCallback. Everything started in connectedCallback must be torn down.

Wiring: UI → Gateway

// Request-response (one-shot)
const result = await this.gateway.request("myFeature.do", {
  name: "test",
  project: this.projectId,
});

// Event subscription (streaming/push)
this._unsub = this.gateway.on("myFeature.update", (data) => {
  this.latestUpdate = data as MyUpdate;
});

// Clean up
override disconnectedCallback() {
  super.disconnectedCallback();
  this._unsub?.();
}
Timeouts. gateway.request() defaults to 30s. For long operations, pass a custom timeout: gateway.request("slow.method", params, 60_000). Constants live in ui/constants.ts.

Adding Styles

ui/styles/ Global CSS (light DOM). Use existing CSS tokens.

  • Use var(--accent) for accent color (#da7756 terracotta)
  • Use var(--font-body) for text, var(--mono) for code
  • Use BEM-style prefixes per component (.ov-card, .chat-sidebar) to avoid collisions

Every feature needs tests. The layer determines the pattern.

Gateway Service Test

tests/gateway/ Import the function, call it, assert. No mocks needed.

import { describe, it, expect } from "vitest";
import { computeSomething } from "../../gateway/services/my-feature.ts";

describe("computeSomething", () => {
  it("returns expected value for normal input", () => {
    expect(computeSomething("hello")).toBe(42);
  });

  it("handles edge case", () => {
    expect(computeSomething("")).toBe(0);
  });
});

Server / WS Handler Test

tests/gateway/server.test.ts Mock services, test dispatch and response shape.

import { handleWsRequest } from "../../gateway/server.ts";
import { requestFrame } from "../../gateway/protocol/frames.ts";

it("myFeature.do returns result", async () => {
  const sent: string[] = [];
  const ws = { send: (d: string) => sent.push(d) } as any;

  await handleWsRequest(ws, requestFrame("r1", "myFeature.do", { name: "test" }));

  const res = JSON.parse(sent[0]);
  expect(res.type).toBe("res");
  expect(res.result.something).toBeDefined();
});

UI Component Test

tests/ui/ Create element, set properties, await render, query DOM.

import { describe, it, expect, beforeEach } from "vitest";
import "../../ui/views/my-feature.ts";
import type { CockpitMyFeature } from "../../ui/views/my-feature.ts";
import { renderEl } from "../helpers.ts";

describe("cockpit-my-feature", () => {
  beforeEach(() => { document.body.innerHTML = ""; });

  it("renders data when provided", async () => {
    const el = document.createElement("cockpit-my-feature") as CockpitMyFeature;
    el.data = { title: "Test", value: 42 };
    await renderEl(el);
    expect(el.querySelector("h2")?.textContent).toBe("Test");
  });

  it("renders skeleton when data is null", async () => {
    const el = document.createElement("cockpit-my-feature") as CockpitMyFeature;
    el.data = null;
    await renderEl(el);
    expect(el.querySelector(".skeleton")).toBeTruthy();
  });
});

Integration Tests

tests/gateway/integration.test.ts Let the real server → CLI chain run, mock only child_process.spawn at the OS boundary.

When do you need an integration test? When the feature wires server.ts to claude-cli.ts — e.g., passing params through to spawn args, processing stdout events into WS frames, or the control protocol round-trip. If your feature is purely within one layer, unit tests are sufficient.

See testing.html for the full integration test architecture, spawn mock details, and the test file map.

5. Deploy

ClaudeCockpit runs locally. "Deploying" means: verify, commit, push, update docs.

Pre-Push Checklist

  1. npm run ci — typecheck + tests + build. If this passes, GitHub Actions CI will too.
  2. Visual check — npm run dev, open localhost:5173, verify the feature works with real session data.
  3. Test count — if you added tests, update the count in testing.html, roadmap.html, and CLAUDE.md.
  4. Changelog — add an entry in changelog.html under the current version (Added / Fixed / Changed / Removed sections).
  5. Roadmap — if a checklist item was completed, toggle check--todo → check--done with checkmark.
  6. Commit — concise message summarizing what and why. Match the style in git log --oneline.

CI Pipeline

GitHub Actions runs on every push to main and on pull requests. Three stages, same as npm run ci:

1. tsc --noEmit          # Type errors block the build
2. vitest run             # Test failures block the build
3. vite build             # Import errors and missing modules caught here

Common Gotchas

Lit decorators need tsconfig flags
useDefineForClassFields: false and experimentalDecorators: true are required. Without them, @property silently breaks and components won't re-render.
JSONL concatenation
Claude Code sometimes writes two JSON objects on one line (missing trailing newline). The parser handles this via splitConcatenatedJson(), but be aware when debugging JSONL issues.
Streaming dedup by message ID
Multiple JSONL lines can share the same message.id (streaming chunks). convertToMessages() deduplicates — the last chunk wins. If messages look duplicated, check the ID.
stdin prompts, not CLI args
Prompts are sent via stdin as JSON, not as CLI arguments. This avoids ARG_MAX limits on long prompts.
Resume skips --model
When resuming (claude -r <id>), buildArgs omits --model. The resumed session keeps its original model. This was a real bug caught by integration tests.
Light DOM = global CSS namespace
All components share one CSS namespace. Use BEM-style prefixes per component (.ov-card, .chat-sidebar) to avoid selector collisions.
