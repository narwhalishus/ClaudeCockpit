ClaudeCockpit

Personal session cockpit for Claude Code — browse sessions, resume conversations, stream live responses.
Built with Lit + Vite + TypeScript. Borrows infrastructure from OpenClaw, designs fresh for Claude Code.

Design Principles

1
Design for Claude Code, not OpenClaw. Borrow infrastructure patterns (Lit components, WebSocket protocol, CSS tokens) but build the information architecture around Claude Code's mental model: projects, subagents, plan mode, tasks, permissions, memory.
2
Personal tool, not a product. No themes, i18n, onboarding, multi-user, or feature flags. Lean and opinionated for a single power user on macOS.
3
Read real data first, control later. Phase 1 is read-only (parse existing JSONL files). Phase 2 adds live interaction. Don't over-engineer the gateway before we know the data shape.

Design Decisions Log

Chat = Session Cockpit (decided 2026-03-31)

The Chat tab is the primary interaction surface, not a utility. Each conversation has a persistent session identity. The two-panel layout — session sidebar on the left, conversation on the right — lets you manage long-lived sessions (like "personal finance app research" or "home renovation planning") and return to them over time.

  • Sessions are scoped to projects (directories). Project switcher controls which sessions appear.
  • New sessions start with claude -p; resuming uses claude -r <id>
  • Project memory (CLAUDE.md, memory/) persists across sessions — the bridge between ephemeral conversations
  • Multiple sessions per project exist for: context window management, parallel workstreams, different scopes, different entrypoints
  • Subagents are inlined in parent session (not separate JSONL files) — shown as collapsible blocks

Tech Stack

  • UI: Lit 3 — Web Components with reactive properties
  • Tooling: Vite 6 — dev server (ESM native) + Rollup bundler
  • Language: TypeScript 5 — strict mode, useDefineForClassFields: false for Lit compatibility
  • Gateway: Node.js HTTP + WebSocket server (ws package, req/res/event frame protocol)
  • Testing: Vitest + jsdom + @open-wc/testing
  • Reference: OpenClaw (MIT) at ~/code/openclaw/ — infrastructure patterns only

Key Data Insights

JSONL Session Format

Each ~/.claude/projects/<encoded-path>/<session-id>.jsonl contains newline-delimited JSON:

{
  "type": "user" | "assistant" | "file-history-snapshot" | "system" | "agent-name" | "custom-title",
  "sessionId": "uuid",
  "timestamp": "ISO-8601",
  "cwd": "/Users/bryao/code/...",
  "entrypoint": "cli" | "claude-vscode",
  "version": "2.1.87",
  "message": {
    "role": "user" | "assistant",
    "content": "string" | [{ type, text, ... }],
    "model": "claude-opus-4-6",
    "id": "msg_...",           // same ID across streaming chunks
    "usage": { input_tokens, output_tokens, cache_read_input_tokens, ... }
  },
  "isSidechain": false,        // branched conversation
  "sourceToolAssistantUUID": "..." // links to parent tool_use
}

Subagent Detection

  • Subagents do not create separate JSONL files — they're inlined in the parent session
  • tool_use blocks with name: "Agent" mark subagent spawns
  • sourceToolAssistantUUID on "user" messages = tool result fed back from assistant
  • type: "agent-name" lines record the session's display name

Project Encoding

Directory -Users-bryao-code-myproject decodes to /Users/bryao/code/myproject (dashes → slashes).
