Test Layers

Each layer catches different classes of bugs. They complement each other.

Layer Tool Runs When What It Catches
Unit tests vitest + jsdom npm test CI Logic bugs, component rendering, API contracts, frame serialization
Integration tests vitest + spawn mock npm test CI Cross-boundary bugs (server → CLI arg wiring, event pipeline, control protocol)
AI visual verification Agent Browser / Playwright MCP During dev Rendering issues, layout breakage, UX regressions

Running Tests Locally

npm test                  # All tests (unit + integration)
npm run test:integration  # Integration tests only
npm run test:watch        # Watch mode for development
npm run typecheck         # TypeScript type check (tsc --noEmit)
npm run ci                # Full local pipeline: typecheck + test + build

npm run ci is the local equivalent of the GitHub Actions pipeline. Run it before pushing to catch issues early.

CI Pipeline

GitHub Actions runs on every push to main and on pull requests. Config: .github/workflows/ci.yml

1
Type check — npx tsc --noEmit. Catches type errors without emitting files. Fails the build if any type is wrong.
2
Tests — npm test (vitest run). Runs all unit and integration tests. Any test failure blocks the build.
3
Build — npm run build (vite build). Verifies the production bundle compiles. Catches import errors and missing modules.

What's NOT in CI (and why)

  • E2E browser tests — Gateway reads ~/.claude/projects/ and spawns claude -p. Neither exists in a GitHub Actions runner.
  • Remote deployment — Gateway must run locally. SPA is non-functional without it.

Integration Test Architecture

File: tests/gateway/integration.test.ts

Integration tests let the real request chain run — handleWsRequest → startChat → ClaudeProcess.buildArgs → spawn — mocking only at the OS boundary:

Unit tests (server.test.ts) Integration tests ───────────────────────── ────────────────────── handleWsRequest [REAL] handleWsRequest [REAL] | | startChat [MOCKED] startChat [REAL] | ClaudeProcess [REAL] | buildArgs [REAL] | spawn [MOCKED] | PassThrough fake process

The spawn mock

Only gateway/services/claude-cli.ts imports child_process. The mock replaces spawn with a factory that returns a controllable fake process:

const proc = Object.assign(new EventEmitter(), {
  stdout: new PassThrough(),  // feed NDJSON to simulate Claude output
  stderr: new PassThrough(),  // feed error text
  stdin:  new PassThrough(),  // capture prompt + control_response writes
  killed: false,
  kill:   vi.fn(),
  pid:    12345,
});

This lets tests feed data through stdout (simulating Claude's NDJSON stream), verify events arrive on the WebSocket, and inspect what gets written to stdin (prompt format, control responses).

What this catches

The model-during-resume bug is the motivating example. Server.ts passed model to startChat, and buildArgs correctly omitted --model when sessionId was set. Unit tests couldn't catch this because they mocked the service boundary between server and claude-cli. Integration tests verify the actual spawn args.

Browser Verification Tools

These are development-time tools used within Claude Code conversations, not automated tests. They don't run in CI.

Agent Browser — Visual Verification
Quick smoke checks after UI changes. Accessibility-tree approach, ~10x more token-efficient than pixel screenshots.

Use for: "Does the chat view render three panels?", "Do the 6 stat cards appear?", "Does the thinking block collapse?"
Playwright MCP — Deep Debugging
Escalate when something looks wrong and you need specifics. Full browser control via MCP tools.

Use for: browser_snapshot (DOM tree), browser_console_messages (JS errors), browser_network_requests (WS frames), browser_take_screenshot.

Recommended workflow

1. Edit code
2. npm test                          ← fast, catches logic + integration bugs
3. Agent Browser: visual verify      ← catches rendering/wiring issues
4. If wrong -> Playwright MCP        ← debug the DOM/network/console
5. Commit + push                     ← triggers CI

Deployment

ClaudeCockpit runs locally. The gateway reads ~/.claude/projects/ JSONL files and spawns claude -p subprocesses — both require a local macOS environment.

npm run dev                    # Development: Vite :5173 + gateway :18800
npm run build && npm run preview   # Production-like local serving

No cloud deployment is needed. The SPA could go to Vercel, but it's non-functional without the local gateway. If remote access is wanted later, a Vercel SPA deploy + SSH tunnel to the gateway is the simplest path.
