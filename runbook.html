Feature Development Workflow

End-to-end process for adding a feature. Every step matters.

1
Orient
Read the relevant source. Understand what exists before changing it.
2
Implement
Gateway service → server wiring → UI component. Bottom-up.
3
Test
Unit tests per layer. Integration test if cross-boundary.
4
Verify
npm run ci locally. Visual check in browser.
5
Ship
Commit, push, update changelog & roadmap.
Bottom-up ordering matters. Build the service logic first (pure functions, easy to test), then wire it into the server (HTTP or WS), then consume it from the UI. Each layer is testable in isolation before the next one depends on it.

Adding a Gateway Service

gateway/services/ Services are functional modules. No classes, no singletons — just exported pure functions with private config.

Pattern

// gateway/services/my-feature.ts

// 1. Types (if needed beyond what's in gateway/types.ts)
export interface MyConfig { /* ... */ }

// 2. Private config / lookup tables
const CONFIG_TABLE: MyConfig[] = [ /* ... */ ];

// 3. Exported pure functions
export function computeSomething(input: string): number {
  // ...
}

export function formatResult(data: unknown): string {
  // ...
}
Keep it pure. No filesystem reads, no process spawning, no side effects in service functions where possible. That's what makes them trivially testable.

Adding an HTTP Endpoint

gateway/server.ts HTTP endpoints live in the handleHttp function. Route on path, extract query params, call a service, return JSON.

Pattern

// In handleHttp():
if (path === "/api/my-feature") {
  const projectId = url.searchParams.get("project") ?? undefined;
  const result = await myServiceFunction(projectId);
  json(res, result);
}

Checklist

  • Add the if (path === ...) branch in handleHttp()
  • Use json(res, data) for success, json(res, { error: "..." }, 500) for errors
  • Add corresponding test in tests/gateway/server.test.ts
  • If the frontend uses it, add proxy entry in vite.config.ts (already proxies /api)

Adding a WebSocket Method

gateway/server.ts WS methods are dispatched via the wsHandlers record. Three steps: handler function, dispatch entry, param extraction.

Pattern

// 1. Handler function (above the dispatch table)
async function handleMyFeature({ ws, id, p }: WsContext) {
  const projectId = getString(p, "project");
  const name = requireString(p, "name");
  if (!name) {
    send(ws, errResponse(id, "INVALID_PARAMS", "name is required"));
    return;
  }

  const result = await myServiceFunction(name, projectId);
  send(ws, okResponse(id, result));
}

// 2. Register in the dispatch table
const wsHandlers: Record<string, (ctx: WsContext) => Promise<void>> = {
  // ... existing handlers ...
  "myFeature.do":  handleMyFeature,  // <-- add here
};

Param Helpers

HelperReturnsUse When
getString(p, key)string | undefinedOptional string param
getNumber(p, key)number | undefinedOptional number param
requireString(p, key)string | nullRequired param — null means missing/empty

Pushing Events (server → client)

For streaming or notifications, use event frames instead of responses:

// Import from protocol/frames.ts
import { eventFrame } from "../protocol/frames.ts";

// Push to a specific client
send(ws, eventFrame("myFeature.update", { progress: 42 }));

// Push to ALL connected clients (rare — used for chat.close)
for (const client of wss.clients) {
  if (client.readyState === WebSocket.OPEN) {
    send(client, eventFrame("myFeature.update", data));
  }
}

Adding a UI Component

ui/views/ or ui/components/ Lit web components in light DOM. All styling via global CSS in ui/styles/.

Pattern

// ui/views/my-feature.ts
import { LitElement, html, nothing } from "lit";
import { customElement, property, state } from "lit/decorators.js";

@customElement("cockpit-my-feature")
export class CockpitMyFeature extends LitElement {
  // Light DOM — no Shadow DOM, uses global CSS
  protected override createRenderRoot() { return this; }

  // Input props (set by parent component)
  @property({ type: Object }) data: MyData | null = null;

  // Internal state (not reflected to attributes)
  @state() private _loading = false;

  // Lifecycle
  override connectedCallback() {
    super.connectedCallback();
    // Subscribe to events, start timers, etc.
  }
  override disconnectedCallback() {
    super.disconnectedCallback();
    // Clean up subscriptions, timers
  }

  // Render
  override render() {
    if (!this.data) return html`<div class="skeleton"></div>`;

    return html`
      <section class="my-feature">
        <h2>${this.data.title}</h2>
        ${this._loading
          ? html`<span>Loading...</span>`
          : nothing}
      </section>
    `;
  }
}

Key Conventions

1
Light DOM always. createRenderRoot() { return this; } — all components render into the main document so global CSS applies. No Shadow DOM.
2
@property for inputs, @state for internal. Both trigger re-renders. Properties are set by parent; state is private to the component.
3
Use nothing not empty strings. Lit's nothing sentinel removes content entirely. ${condition ? html`...` : nothing} is the pattern for conditional rendering.
4
Clean up in disconnectedCallback. Timers, subscriptions, event listeners — anything started in connectedCallback must be torn down.

Wiring End-to-End

How data flows from a gateway service through WS to the UI.

Gateway Service Server (WS) UI Client UI Component ────────────── ─────────── ───────── ──────────── Pure function → wsHandler calls it → gateway.request() → @property data in services/ sends okResponse() returns Promise triggers render() or eventFrame() or on() callback

UI Side: Making Requests

// In a Lit component (e.g., ui/app.ts or ui/views/chat.ts)
// The gateway client is usually owned by the app shell and passed down,
// or instantiated directly in the component.

// Request-response (one-shot)
const result = await this.gateway.request("myFeature.do", {
  name: "test",
  project: this.projectId,
});

// Event subscription (streaming/push)
this._unsub = this.gateway.on("myFeature.update", (data) => {
  this.latestUpdate = data as MyUpdate;
});

// Unsubscribe in disconnectedCallback
override disconnectedCallback() {
  super.disconnectedCallback();
  this._unsub?.();
}
Timeouts. gateway.request() defaults to 30s. For long operations (summaries, large data), pass a custom timeout: gateway.request("slow.method", params, 60_000). Constants live in ui/constants.ts.

Type System

Types are intentionally duplicated across two files. This is a deliberate boundary, not tech debt.

FileModelsExamples
gateway/types.ts JSONL on-disk format, raw content blocks, API response shapes RawSessionLine, RawContentBlock, RawUsage, SessionSummary
ui/types.ts Simplified shapes for rendering, plus UI-only state ChatMessage (adds streaming), SessionMessage, ToolApprovalEvent

When to add types where

  • New JSONL field or Claude Code data shape? → gateway/types.ts
  • New API response shape from a gateway endpoint? → gateway/types.ts
  • New UI rendering state or component prop type? → ui/types.ts
  • Shared constant (model list, timeouts)? → ui/constants.ts

Testing

Tests live in tests/ mirroring source structure. Three patterns depending on what you're testing.

Gateway Service Test

tests/gateway/ Import the function, call it, assert the result. No mocks needed for pure services.

// tests/gateway/my-feature.test.ts
import { describe, it, expect } from "vitest";
import { computeSomething } from "../../gateway/services/my-feature.ts";

describe("computeSomething", () => {
  it("returns expected value for normal input", () => {
    expect(computeSomething("hello")).toBe(42);
  });

  it("handles edge case", () => {
    expect(computeSomething("")).toBe(0);
  });
});

Server / WS Handler Test

tests/gateway/server.test.ts Uses mocked services. Import handleWsRequest, pass a fake WebSocket and RequestFrame, assert what was sent back.

// Pattern from server.test.ts:
import { handleWsRequest } from "../../gateway/server.ts";
import { requestFrame } from "../../gateway/protocol/frames.ts";

it("myFeature.do returns result", async () => {
  const sent: string[] = [];
  const ws = { send: (d: string) => sent.push(d) } as any;

  await handleWsRequest(ws, requestFrame("r1", "myFeature.do", { name: "test" }));

  const res = JSON.parse(sent[0]);
  expect(res.type).toBe("res");
  expect(res.result.something).toBeDefined();
});

UI Component Test

tests/ui/ Create the element, set properties, await renderEl(), query the DOM.

// tests/ui/my-feature.test.ts
import { describe, it, expect, beforeEach } from "vitest";
import "../../ui/views/my-feature.ts";
import type { CockpitMyFeature } from "../../ui/views/my-feature.ts";
import { renderEl } from "../helpers.ts";

describe("cockpit-my-feature", () => {
  beforeEach(() => { document.body.innerHTML = ""; });

  it("renders data when provided", async () => {
    const el = document.createElement("cockpit-my-feature") as CockpitMyFeature;
    el.data = { title: "Test", value: 42 };
    await renderEl(el);

    expect(el.querySelector("h2")?.textContent).toBe("Test");
  });

  it("renders skeleton when data is null", async () => {
    const el = document.createElement("cockpit-my-feature") as CockpitMyFeature;
    el.data = null;
    await renderEl(el);

    expect(el.querySelector(".skeleton")).toBeTruthy();
  });
});

Test Helpers (tests/helpers.ts)

HelperWhat It Does
renderEl(el)Appends element to document.body, awaits Lit's updateComplete, returns element

Integration Tests

tests/gateway/integration.test.ts For features that cross the server → CLI boundary. These let real code run through the full chain, mocking only child_process.spawn.

When do you need an integration test? When the feature involves wiring between server.ts and claude-cli.ts — e.g., passing params through to spawn args, processing stdout events into WS frames, or the control protocol round-trip. If your feature is purely within one layer, unit tests are sufficient.

See testing.html for the full integration test architecture diagram and spawn mock details.

Adding Styles

ui/styles/ All CSS is global (light DOM). Add styles for new components to the appropriate stylesheet.

  • New view? → Add a section in the relevant stylesheet or create a new one and import it in index.html
  • Reusable pattern? → Use existing CSS tokens from :root variables
  • Accent color: #da7756 (Claude terracotta) — use var(--accent)
  • Fonts: var(--font-body) for text, var(--mono) for code

Pre-Push Checklist

Run through this before every push. The CI will catch most of it, but catching locally is faster.

  1. npm run ci — typecheck + tests + build. If this passes, CI will too.
  2. Visual check — npm run dev, open localhost:5173, verify the feature works with real data.
  3. Test count — if you added tests, update the count in testing.html, roadmap.html, and CLAUDE.md.
  4. Changelog — add an entry in changelog.html if behavior changed.
  5. Commit message — concise summary of what and why. Look at git log --oneline for the style.

Common Gotchas

Lit decorators need tsconfig flags
useDefineForClassFields: false and experimentalDecorators: true are required. Without them, @property silently breaks and components won't re-render.
JSONL concatenation
Claude Code sometimes writes two JSON objects on one line (missing trailing newline). The parser handles this via splitConcatenatedJson(), but be aware when debugging JSONL issues.
Streaming dedup by message ID
Multiple JSONL lines can share the same message.id (streaming chunks). convertToMessages() deduplicates these — the last chunk wins. If messages look duplicated, check the ID.
stdin prompts, not CLI args
Prompts are sent via stdin as JSON ({ type: "user", message: { role: "user", content: "..." } }), not as CLI arguments. This avoids ARG_MAX limits on long prompts.
Resume skips --model
When resuming a session (claude -r <id>), buildArgs intentionally omits --model. The resumed session keeps its original model. This was a real bug caught by integration tests.
Light DOM means global CSS conflicts
All components share one CSS namespace. Use BEM-style prefixes per component (.ov-card, .chat-sidebar) to avoid selector collisions.
